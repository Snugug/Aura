AURA

Responsive Done Easy

Building responsive websites is hard. From grid calculations to unit conversions to remembering just where you put that last breakpoint, there’s a lot of moving parts that need to be taken care of in order for your website to come out right. Aura aims to simplify that.

Aura’s goal is to help get you set up and help you keep organized, and baring those two things, get the hell out of your way. Built on top of Susy, Aura will help you build out a fluid grid based and some basic breakpoints based on your content without any math needed. Designed to make your life easier, Aura will help you build your website from a Mobile First perspective, with helpers for sidebars and a breakpoint API to keep your media queries in check.

Installation and Basic Setup

First, install the Aura gem:

(sudo) gem install compass-aura

If you are creating a new project, require Aura:

compass create <my-project> -r aura

If you are initializing a project, require Aura as well:

compass init -r aura

If you have a project already and would like to use Aura, add the following to the top of your config.rb file:

require 'aura'

In your Sass file, import Aura:

@import 'aura';

Now here is where Aura differs from some other Compass extensions (at least until I experiment a little bit more and see if I can solve this; it’s probably very easy, I just need to go ahead and do it). Because Aura builds your grid based on your input, there is a mixin you need to call in order to get Aura set up. First, set all of your Aura variables (see the Basic API), then call the aura-setup mixin:

@include aura-setup;

Finally, you need to hook your HTML structure up to Aura. There are four pieces of Aura that you need to setup: the container, the main content region (called the measure), the primary sidebar, and the secondary sidebar (currently, Aura only supports a maximum of two sidebars; for more information, see the Sidebar API). The IDs below are used for demonstration purposes; use your own!

#page {
    @extend %aura-container;
  }

  main-content {
    @extend %measure;
  }

  sidebar-first {
    @extend %primary-sidebar;
  }

  sidebar-second {
    @extend %secondary-sidebar;
  }

Compile your CSS, and you’ve got a website!

Basic API

There are a handful of basic setup options that you can set to customize your grid without needing to do any math (it’s really fun to play with these, have something like Live Reload running, and watch your whole grid rearrange itself. Try it!). Do not set any of Susy’s variables (except as noted), Aura takes care of that for you.

$body-font-size: The base font size to use for your site, generally the font size your body font looks best at. This will be applied to the HTML element, making using em/rem units super easy. Use any unit you want, px, em, pt, or %, they will all be converted to em. Defaults to 16px.

$total-columns: Identical to Susy’s total columns variable, with the exception that, depending on your sidebar options, this may be changed for you. Defaults to 12.

$main-content-cols: The number of columns you want your main content area to be. Defaults to 9.

$font-scale: Toggle if you want font scaling turned on. Font scaling will take your $body-font-size and scale it using Modular Scale up two and down two scales. This is applied to your HTML element, making using em/rem units super easy, doubly so as everything will then scale with your base font size. If you turn off Font Scaling, your breakpoints are no longer tied to your content, and as such you’re going to need to play with $responsive-ratio to get your single column break correct. This site uses Aura's Font Scaling. Defaults to true.

$obj-scale: Toggle whether or not you want objects to scale (such as img{ max-width: 100%}). Defaults to true.

$sidebars: Controls your sidebars. See Sidebar API for full information. Defaults to one sidebar on the right (1, ‘right).

$single-column: Displays your site in its single column form. Useful for Mobile First design. Defaults to false.

$breakpoints: A comma separated list of named breakpoint to use throughout your files. See Media Query API for full information. Defaults to false.

Advanced API

The Advanced API items control the math that takes care of generating your grid and some suggested break points. It’s generally best not to mess with these unless you want to get into the nitty-gritty of the math used in building your grid.

$gutter-to-col: The ratio of gutter width to column width, Defaults to 1 to 4 (1, 4).

$alphabet-count: Number of characters in the alphabet you’re using. Used to calculate typographic measure for the web. Defaults to 26.

$measure-width: Mathematical calculation of the number of characters in the measure. Not a direct character for character count, as web measurements are based on the width of the m character (ems) whereas non-monospaced characters fluctuate in width, making the exact result slightly different than ideal measure width. The default more or less ideally mimics the ideal measure when used with font scaling. Defaults to $alphabet-count time the $fourth modular scale variable (4/3)

$lh-addition: The amount, in ems, to add to font-size to get line height. This will probably be changing. Defaults to .3em.

$responsive-ratio: The ratio that the suggested breakpoints and font scaling is built off of. If Font Scaling is turned on, default to $major-third (5/4). If Font Scaling is turned off, defaults to $golden-ratio (1.618).

$width: If you know the max size you want your grid to be, instead of having Aura build your grid for you based on your content, you can set this to a max width and Aura will calculate the grid for that width. This is not the recommended approach, as it detaches your design from your content, and your content should be driving your design. This will turn off Font Scaling. Defaults to false.

Sidebar API

Aura aids you in building out your sidebars from a Mobile First perspective, starting with a single column and adding on layout using media queries as is determined by ideal measure and available screen real estate with Font Scaling on, and by your Responsive Ratio when Font Scaling is turned off. Let’s take a look at how you can build and position your sidebars using the Sidebar API.

A word of note about the Sidebar API. The API was designed with a few assumptions in mind, the first being that you are working Mobile First and the second being that your HTML Structure is clean and semantic, having a rough DOM structure as follows:

  -html
  --body
  ---page wrapper
  ----header
  ----content
  ----primary sidebar
  ----secondary sidebar
  ----footer

If this is not how your DOM structure roughly looks, I do not know, in fact am fairly sure, that the Sidebar API will not work for you. With that in mind:

There are four parts to the Sidebar API, the number of sidebars you want, where you want them positioned, how many columns you want the primary sidebar to be, and how many columns you want your secondary sidebar to be. For the purposes of Aura, the Primary Sidebar is always going to be the single sidebar if you have one sidebar and the leftmost sidebar if you have two sidebars, so the first of two sidebars if you have both sidebars on the left or on the right, and the left sidebar if you have sidebars on either side of your content area. The Secondary Sidebar, on the other hand, is only used if you have two sidebars and is always the rightmost sidebar, so the second of two sidebars if you have both sidebars on the left or on the right, and the right sidebar if you have sidebars on either side of your content.

No Sidebars

If you do not want any sidebars (aka are building a single column website), you can write:

$sidebars: 0, 'none';

One Sidebar

If you would like to have one sidebar, you would set the first option to 1 and the second option to the side you want them on, left or right:

$sidebars: 1, 'right';
  $sidebars: 1, 'left';

The styling gets applied to the Primary Sidebar extendable. The width for this sidebar is determined by subtracting $main-content-cols from $total-columns. If you know exactly how many columns you want your sidebar to take up, you can set a third option, number of columns. If your new column count would exceed $total-columns, that is increased so that your sidebar and your content will fit within your grid. So, say I want one sidebar on the right and I want it to span four columns, I would write:

$sidebars: 1, 'right', 4;

Two Sidebars

Aura will also assist you in placing two sidebars, either both on the left

$sidebars: 2, 'left';

both on the right

$sidebars: 2, 'right';

or one on each side

$sidebars; 2, 'both';

The widths for the sidebars are determined by dividing the difference between $main-content-cols and $total-columns in half. If it cannot be divided evenly in half, the amount is rounded down and the extra column is given to the Primary Sidebar, so if the difference was 4 columns, each sidebar would be 2 columns wide, whereas if the difference was 3 columns, the Primary Sidebar would be 2 columns wide and the Secondary Sidebar would be 1 column wide. In much the same way you could determine the width of one sidebar, you can for two. If you add in a single option, that will set both sidebars to that width, so

$sidebars: 2, 'right', 4;

would give you two sidebars, both on the right, both spanning 4 columns. You can also choose the sizes independently with a fourth option, so

$sidebars: 2, 'both', 4, 3;

would give you two sidebars, one on either side of your content, with the left sidebar (the Primary Sidebar) spanning four columns and the right sidebar (the Secondary Sidebar) spanning three columns.

Media Query API

Aura aids you in building out your media queries, be them from tv, print, screen, anything really. There are three pieces to Aura’s Media Query API: the respond-to mixin, the $breakpoints variable, and the media-query mixin. Each will help you write your media queries in their own way. Each method will produce media queries that will bubble out just like a hand written media query.

Media Query Mixin

A simple and elegant solution not wanting to write out the whole media query syntax every time. The mixin takes one parameter with two additional optional parameters. The needed parameter is the value you want for your media query, most commonly a number, while the two optional parameters are the $operator, most commonly min or max width, and the $query, most commonly screen. The defaults for $operator is min-width and the default for $query is screen.

To use the Media Query mixin, call the mixin with the values you want. For instance, if you wanted a min-width screen query at 20em, you would write:

@include media-query(20em) {}

with your selectors and properties within the curly braces. If you wanted a max-width screen media query at 450px, you would write:

@include media-query(450px, 'max-width'){}

If you wanted a min-width tv query at 30em, you would write:

@include media-query(30em, $query: 'tv'){}

You can also pass a list into the media query mixin with options in the order of Size, Operator, and Query, and that will work too, so you could do something like:

$tv: 30em 'max-width' 'tv'; 
  @include media-query($tv){}

Breakpoint Variable and the Respond-To Mixin

As Mason Wendell describes in his Sass Way blog post, a mixin that you can call using named breakpoints allows you to reuse breakpoints easily and, more importantly, keep track of your breakpoints in a much simpler manner than remembering each numbered breakpoint. Aura builds out four suggested breakpoints for you, small, medium, large, and full, and you can use them by including the respond-to mixin with one of them.

@include respond-to('small') {}

The two breakpoints you’ll probably use the most are small, which is where your single column becomes multiple columns, and full, which is at full width of your website. Medium and large will probably be used less often, but are essential for Font Scaling, and may be useful if you have it turned off, but get generated anyway as a matter of course.

But, you say, what if I want to define custom breakpoints? Well then, as it turns out, the respond-to mixin in Aura is actually a slightly modified version of my Variable Driven Respond-To Mixin, allowing you to set media queries as parts of a list variable and then use them with respond-to! Each breakpoint in the list requires, at a minimum, the name of the breakpoint and the value of the breakpoint; like the Media Query mixin, this assumes your operator is min-width and query is screen; the big difference is if you want to change your query, you also need to define your operator, you can’t just call the query variable.

Well that’s a mouthful, what does it actually mean? Well, I think the code example from the gist will explain it all:

// Four breakpoints: foo, bar, baz, and qux.
//
// Foo assumes the defaults of min-width and screen. Foo will gen @media screen and (min-width: 500px)
// Bar changes min-width to max, but assumes screen as well. Bar will gen @media screen and (max-width: 700px)
// Baz wants to use min-width, but change media type, so min-width needs to be re-declare. Baz will gen @media tv and (min-width: 700px)
// Qux goes all out and has a full unique media query. Qux will get @media tv and (max-width: 900px)


$breakpoints: ‘foo’ 500px, ‘bar’ 700px ‘max-width’, ‘baz’ 700px ‘min-width’ tv, ‘qux’ 900px ‘max-width’ tv;

// Let’s call respond-to!

waldo {
  background: red;

  @include respond-to(‘foo’) {
    background: green;
  }

  @include respond-to(‘bar’) {
    background: yellow;
  }

  @include respond-to(‘baz’) {
    background: purple;
  }

  @include respond-to(‘qux’) {
    background: orange;
  }
}

// Our Output:

waldo {
  background: red;
}

@media screen and (min-width: 500px) {
  #waldo {
    background: green;
  }
}

@media screen and (max-width: 700px) {
  #waldo {
    background: yellow;
  }
}

@media tv and (min-width: 700px) {
  #waldo {
    background: purple;
  }
}

@media tv and (max-width: 900px) {
  #waldo {
    background: orange;
  }
}
